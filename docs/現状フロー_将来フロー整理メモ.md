# 現状フロー / 将来フロー整理メモ

## 1. 目的

本メモは、claude_orchestrator における現在の role / workflow の実態を整理し、  
将来的に目指すフローとの差分を見える化するための整理資料である。

ここでいう「現状」は、以下の実装・ルールを確認した内容を基準とする。

- `docs/workflow_rules.md`
- `src/claude_orchestrator/core/workflow.py`
- `src/claude_orchestrator/application/usecases/advance_task_usecase.py`
- `src/claude_orchestrator/application/usecases/run_task_usecase.py`
- `src/claude_orchestrator/application/usecases/validate_report_usecase.py`
- `src/claude_orchestrator/application/usecases/generate_next_task_proposals_usecase.py`

---

## 2. 現状の基本構造

現状の実行 workflow は、以下の 4 role を中心に構成される。

- `task_router`
- `implementer`
- `reviewer`
- `director`

加えて、実行 workflow の外側に補助 role として `planner` が存在する。

### 2-1. 現状の位置づけ

- `task_router` は実行前整理役
- `implementer` は実装 / 実行担当
- `reviewer` はレビュー担当
- `director` は最終判断担当
- `planner` は completed task をもとに次 task 候補を提案する補助 role

---

## 3. 現状フロー

### 3-1. 実行 workflow

現状の task 実行フローは以下である。

```text
task_router
  ↓
implementer
  ↓
reviewer
  ↓
director
  ├─ approve → completed
  ├─ revise  → task_router に戻る（cycle+1）
  └─ stop    → stopped

## 3-2. 異常系
task_router(blocked)            → blocked
implementer(blocked/need_input) → blocked
reviewer(blocked)               → blocked
3-3. revise 時の再開点

director が revise を返した場合、再開点は implementer ではなく task_router である。

これは次サイクル開始時に、task の再整理と skill 配布をやり直すためである。

4. 現状の role ごとの責務
4-1. task_router

役割:

task の実行前整理

task_type の分類

risk_level の判断

role_skill_plan の決定

initial_execution_notes の整理

制約:

実装しない

reviewer / director を兼ねない

state.json を更新しない

補足:

task_router の report は advance 時に task.json へ反映される

4-2. implementer

役割:

実装または実行作業を行う

実際に使用した skill を used_skills に記録する

ステータス:

done

blocked

need_input

補足:

done の場合のみ reviewer へ進む

blocked / need_input は停止扱い

4-3. reviewer

役割:

implementer の結果確認

品質・妥当性確認

director への判断材料整理

decision:

ok

needs_fix

blocked

補足:

ok と needs_fix はどちらも director へ進む

blocked は停止扱い

4-4. director

役割:

最終判断

approve / revise / stop の決定

final_action:

approve

revise

stop

補足:

approve で completed

revise で cycle+1 し task_router へ戻る

stop で stopped

4-5. planner

役割:

completed task を入力に次 task 候補を提案する補助 role

制約:

実行 workflow に参加しない

skill を自動決定しない

task を自動確定しない

state 遷移を決めない

補足:

planner は source task の task/state/report 群と docs 群を参照して proposal を生成する

5. 現状の機械的実行フロー

GUI 自動実行では、各 role ごとに以下の順で進行する。

show-next
↓
Claude 実行
↓
report JSON 保存確認
↓
validate-report
↓
advance
↓
state 更新
↓
次 role へ

このループを、completed / blocked / stopped / next_role=none のいずれかになるまで継続する。

5-1. validate で見ているもの

現状の validate は少なくとも以下を確認している。

report file の存在

schema 一致

task_id 一致

role 一致

cycle 一致

つまり現状でも、Claude の report はかなり厳密に検証されている。

6. 現状の設計上の特徴

現状の特徴は以下である。

role 分離が明確

JSON report ベースで遷移する

Python 側は validate と state 更新を機械的に実施する

Claude 側は role に従って report を出力する

revise 時に task_router へ戻すことで再整理を必須化している

planner は実行系から分離されている

このため、現状はすでに「role 分離された task 実行オーケストレータ」として成立している。

7. 将来フローの方向性

今後は、現状の実行 workflow を土台にしつつ、以下の 3 ライン構成へ発展させる方針が有力である。

標準作業ライン

改善提案ライン

出荷ライン

8. 将来フロー案
8-1. 標準作業ライン

現状の本線を、品質統制を強めたうえで維持する。

task
  ↓
task_router
  ↓
implementer
  ↓
reviewer
  ↓
director
  ↓
release / git_operator
  ↓
completed / github反映
8-2. 改善提案ライン

現状の planner を、より明確に改善提案制度として扱う。

completed task / 現在の課題
  ↓
planner_safe / planner_improvement
  ↓
proposal分類
  - safe
  - improvement
  - challenge
  ↓
review / director 判断
  ↓
採用案のみ task 化
  ↓
標準作業ラインへ投入
8-3. 出荷ライン

実装完了後の Git 操作を、実装 role から分離して扱う。

director approve
  ↓
release / git_operator
  ↓
branch確認
  ↓
commit
  ↓
push
  ↓
必要なら PR 文面生成

10. 将来設計で強めたい点
10-1. 標準作業の厳格化

現状の本線を保ちながら、以下を強める。

role ごとの責務と禁止事項の明文化

必須 docs の整理

role ごとの参照 docs の選別

工程ゲートの厳格化

validation 条件の追加

不適合時の扱いの整理

10-2. 改善提案制度の実装

planner が無難化しすぎないよう、改善提案制度を構造として追加する。

方向性:

safe / improvement / challenge を分ける

攻めの提案は直接実装させない

必ず提案 → 評価 → 承認 → task化 を通す

10-3. Git / GitHub 操作の専用化

実装 role に Git 出荷責務を持たせず、専用 role として分離する。

方向性:

implementer は push しない

reviewer は差分妥当性確認まで

director が出荷可否を判断

release / git_operator が branch / commit / push を担当する

11. 将来像の設計思想

将来像として目指すのは、単なる AI 実行ツールではなく、以下を備えた軽量 QMS 型の開発実行統制システムである。

規範層

実行層

証跡層

統制層

出荷層

思想としては、

標準作業で安定運用する

改善提案制度で攻めを管理する

出荷責務を専用 role に切り出す

AI を賢くするより工程を賢くする

ことを重視する。

12. 現時点の整理結論

現状はすでに、以下が成立している。

task_router → implementer → reviewer → director の明確な実行 workflow

strict な validate → advance のループ

revise 時の task_router 再起点

planner の実行 workflow 外分離

GUI からの非同期自動実行

今後はこれを土台に、

標準作業ラインの厳格化

改善提案ラインの追加

release / git_operator の導入

へ拡張していく方針が自然である。
