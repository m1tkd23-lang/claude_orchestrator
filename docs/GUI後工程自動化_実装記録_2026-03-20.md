# GUI後工程自動化 実装記録 2026-03-20

## 1. 目的

本記録は、claude_orchestrator における GUI ベースの後工程自動化実装について、  
今回区切りまでに実装・確認できた内容を整理し、次工程へ引き継ぐための記録である。

対象テーマは主に以下である。

- planner_safe / planner_improvement の GUI 運用強化
- plan_director の導入
- director approve 後の後工程自動起動
- proposal 採択後の task 自動生成
- 次 task の自動実行
- UI ロック制御
- 停止予約
- task_created 状態対応

---

## 2. 今回到達したフロー

今回時点で、GUI から以下の半自動フローが成立した。

```text
task
  ↓
task_router
  ↓
implementer
  ↓
reviewer
  ↓
director
  └─ approve
        ↓
   planner_safe or planner_improvement
        ↓
   plan_director
        ↓
   GUI に結果表示
        ↓
   人間が「次task作成を承認」
        ↓
   CreateTaskFromPlanDirectorUseCase
        ↓
   新task 自動実行

現時点では、最後の task 作成投入に人間承認を残している。
これは将来の自動承認化を見据えた安全ゲートとして位置づけている。

これにより、実行タブから見たときに、

後工程中でも task 一覧や Claude実行ボタンが押せてしまう問題があったため、
pipeline 全体を busy として扱うロック制御へ修正した。

busy 判定対象:

plan_director_active

pending_auto_planner_after_completion

pending_auto_plan_director_after_planner

waiting_next_task_approval

この修正により、後工程中は以下を触れないようにした。

新規 task 作成欄

Claude 実行ボタン

手動操作ボタン群

planner role 切替

planner proposal 操作群

承認待ちでは、次のみ許可する形に整理した。

次task作成を承認

今回は作成しない

3-6. 停止予約

将来の自動化を見据え、
「現在のタスク作業が終わった時点で停止させる」予約を入れられるようにした。

追加内容:

完了後停止予約ボタン

auto run 中のみ有効

現在 task 完了後に planner 後工程へ進まず停止

この機能により、全自動方向へ進めた後でも、
人間が安全に介入できる設計の土台を用意できた。

3-7. task_created 状態対応

plan_director で adopt 後に task を作成した proposal が、
後から元 task を見直した際に再び承認待ち扱いになって UI がロックされる問題があった。

対応内容:

proposal state に task_created を追加

CreateTaskFromProposalUseCase で task 作成後 state を task_created に更新

ProposalStateStore でも task_created を許可

plan_director result 再読込時に、selected proposal が task_created 済みなら
waiting_next_task_approval に戻さないよう修正

この対応により、過去 task を開いたときの誤ロックを解消した。

3-8. stale pending フラグ修正

次task承認後に auto run を再開しようとすると、
pipeline is already active で止まる問題があった。

原因は _pending_auto_planner_after_completion が stale なまま残っていたこと。
main_window_auto_run.py 側で後工程起動直前に当該フラグを False 化し、
次task 自動実行へ正常に遷移するよう修正した。

6. 今回の設計上の重要ポイント

今回の設計で重要だった考え方は以下。

6-1. 標準ラインと後工程ラインの責務分離

RunTaskUseCase は標準ライン専任

GUI 側は後工程オーケストレーションを担当

これにより、既存の標準ライン実装を大きく崩さず拡張できた。

6-2. planner と plan_director の分離

planner は提案

plan_director は採択

この分離を維持したことで、
改善提案の自由度と task 投入時の安全性を両立しやすくなった。

6-3. 人間承認は最後だけ残す

後工程の自動化を進めつつ、
最終 task 投入のみ人間が承認する構造としたことで、
暴走リスクを低く保ちながら自走化へ近づける形にできた。
