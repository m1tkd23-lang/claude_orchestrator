# completion_definition

## 目的

この文書は、開発対象 repo における「完成」の判断基準を定義する。  
task_router / planner / plan_director は、この文書を参照して以下を判断する。

- 今回の task が完成条件のどこに寄与するか
- 次に埋めるべき不足は何か
- 提案や採択が完成条件に近づくか
- 今は対象外の作業かどうか

---

## 基本方針

- 完成条件は **MVP 到達条件** を優先する
- 追加価値は、MVP を壊さない範囲で扱う
- task 提案や採択では、主線を塞ぐ不足を優先する
- すでに実装済みのものを完成条件として重複提案しない
- GUI / CLI / docs / validation / import/export など、利用価値に関わる不足を具体的に扱う

---

## 完成判定のレベル

### 1. MVP 完成
ユーザーが主要用途を安全に一通り実行できる状態。

### 2. 運用安定化
MVP は成立しており、エラー対応・保守性・運用性を高める段階。

### 3. 将来拡張
現時点では必須ではないが、将来価値の高い改善や拡張。

---

## 主要完成条件

以下は repo ごとに更新して使う。  
各項目は `未着手 / 一部完了 / 完了 / 対象外` のいずれかで考える。

### 主線機能
- 主要データの新規作成ができる
- 主要データの保存と再読込ができる
- 主要データの編集ができる
- 主要利用フローが end-to-end で通る

### 入出力
- 正本データの保存形式が安定している
- 必要な import/export が利用可能
- 既存業務との受け渡し形式に対応している

### GUI / 操作性
- 起動後に主要操作へ迷わず入れる
- 主要編集操作が GUI から実行できる
- 必要な保存・確認・エラー表示がある

### 品質 / 検証
- 最低限の validation がある
- 主要 usecase に対する test がある
- エラー時に壊れにくい

### docs / 運用
- README や利用導線が最低限ある
- 運用上重要な制約が文書化されている

---

## 優先順位づけルール

task_router / planner / plan_director は、完成条件への寄与を以下の順で重く見る。

1. 主線を直接前進させる不足
2. GUI / CLI など利用導線を成立させる不足
3. import/export や validation など運用上重要な不足
4. 保守性・改善価値の高い不足
5. 将来拡張のみを目的とする作業

---

## 提案・採択時の扱い

### task_router
- task が完成条件にどう寄与するかを意識する
- 完成条件との接続が弱い task は、必要なら blocked や粒度再整理を検討する

### planner
- completion_definition の未達項目を優先して proposal を出す
- 完了済み項目を未実装のように再提案しない
- 将来価値だけの提案は、mainline では優先度を誇張しない

### plan_director
- 採択 proposal が completion_definition に近づくかを重視する
- 主線前進量と完成条件への寄与が弱い proposal は過大評価しない

---

## 対象外の扱い

以下は現時点で対象外にしてよい。

- 明確な利用価値がまだない大規模拡張
- MVP 成立前の過度な最適化
- 主要完成条件と無関係な局所改善
- 実装済みだが表現違いで重複して見える作業

---

## 更新ルール

- 開発の進行に応じて見直してよい
- repo の現在地とズレたら更新する
- 抽象論だけでなく、具体的な完成条件に寄せる
- feature_inventory と矛盾しないように保つ
