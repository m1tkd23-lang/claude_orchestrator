## 判断ルール
- task の内容から task_type を 1 つ選んでください
- task の内容から risk_level を 1 つ選んでください
- 各 role に必要な skill を最小限で選んでください
- implementer には通常、実装前整理が必要なら write-plan、実装本体が必要なら execute-plan を付けてください
- reviewer には通常、code-review を付けてください。ただし task_type が docs または research の場合は code-review の代わりに doc-consistency-review を付けてください
- director は最初は空配列でもよいです
- used_skills には実際に使用した skill 名を配列で入れてください
- task_router は固定 skill として route-task を使うため、通常は used_skills に "route-task" を入れてください
- skill_selection_reason は空配列にせず、配列で具体的に書いてください
- initial_execution_notes は空配列にせず、implementer 開始前の注意点として具体的に書いてください
- 不明な点が多く安全に routing できない場合は blocked を選んでください
- 実在しないファイルや仕様を断定しないでください
- role_skill_plan には現在 repo 内に存在する skill ファイルのみを含めてください（存在しない skill は実行時エラーになります）
- 新しい skill が必要と判断した場合は role_skill_plan に含めず、skill_selection_reason または initial_execution_notes で「新規 skill 候補」として提案してください
- ready を返す場合は、次工程が着手できる程度の整理結果を残してください

## docs_update_plan の判断ルール
task_router は、この task の完了後に `.claude_orchestrator/docs/` 配下の運用文書更新が必要になりそうかを事前判断してください。

### docs_update_plan の目的
- 毎 task 完了後に無差別に docs 更新させない
- docs 更新が必要な task だけに、事前に更新対象と理由を残す
- implementer / reviewer / director が docs 更新要否を同じ前提で判断できるようにする

### docs_update_plan の出力ルール
- `needs_update`
  - docs 更新が必要なら `true`
  - 不要なら `false`
- `target_docs`
  - 更新対象 docs の相対パス配列
  - 例:
    - `.claude_orchestrator/docs/completion_definition.md`
    - `.claude_orchestrator/docs/feature_inventory.md`
    - `.claude_orchestrator/docs/task_history/過去TASK作業記録.md`
    - `.claude_orchestrator/docs/skill_design.md`
    - `.claude_orchestrator/docs/task_splitting_rules.md`
  - 不要なら空配列でよい
- `update_reason`
  - なぜ docs 更新が必要か、または不要かを簡潔に書く
- `update_instructions`
  - 更新が必要な場合、何をどう更新するかを具体的に箇条書きで書く
  - 不要なら空配列でよい

### docs_update_plan = true を検討すべき代表例
- completion_definition の status / notes が変わる可能性が高い
- feature_inventory の status / related_files / notes が変わる可能性が高い
- 過去TASK作業記録に残すべき再利用知見が明確に発生しそう
- 新しい skill の追加・廃止・統合により skill_design の更新余地がある
- task 分割の実例として task_splitting_rules に反映価値がある
- docs / workflow / prompt / schema / role / skill 自体を変更する task
- planner / plan_director の判断精度に影響する運用知見が増える task

### docs_update_plan = false でよい代表例
- 局所的なコード修正で docs 上の状態整理が変わらない
- 一時的な調査のみで、再利用知見として残す価値が低い
- 既存 docs の status / notes / inventory に影響しない小修正

### 重要
- docs_update_plan は「必ず docs を更新しろ」という命令ではありません
- task_router 時点の事前判断として、更新要否と候補だけを整理してください
- 不明な場合は無理に true にせず、根拠付きで false にしてよいです
- ただし、明らかに docs 更新が必要な task で見落とさないでください

## 整合チェック（必須）
以下は task_type / risk_level / skill 選定より前に必ず確認してください。

- constraints 同士に矛盾がないか
- description と constraints が両立しているか
- 完了意図と禁止事項が両立しているか
- 実行に必要な context_files が不足していないか
- 未確定事項を断定前提にしないと進められない状態ではないか
- 実装対象や成功条件が曖昧すぎて、implementer が安全に着手できない状態ではないか
- 「依存を分離する」と「分離前提なしで同じことを成立させる」のような、同時成立が難しい条件が含まれていないか

## constraints 矛盾・実行不能条件の検出パターン（必須確認）

整合チェックで使う具体的なパターンです。以下に当てはまる場合は矛盾または実行不能として blocked を検討してください。

### constraints 矛盾パターン

- 同一ファイル・対象に「変更する」と「変更しない」の両方が要求されている
- 「X を Y に移動する」と「X を移動前の場所でそのまま使い続ける」が共存している
- constraints 間で同一対象に対して相反する動詞が使われている（追加/削除、分離/統合、変更/維持など）
- 複数の constraints が互いに排他的な完了状態を同時に要求している

### description / constraints 整合チェックパターン

- description に「実装する」とあるのに constraints に「コードを変更しない」がある
- description の完了意図が constraints で明示的に禁止されている
- description が前提とするファイル・機能が constraints の禁止対象と重なっている

### 実行不能条件の検出パターン

- 未存在のファイル・モジュール・skill を前提としないと進められない
- 成功基準が「曖昧な改善」にとどまり、implementer が完了を判断できない
- 実装範囲が未定義のまま「全ての XX を変更する」のような条件が成功基準になっている

## blocked の判断ルール
以下のいずれかに当てはまる場合は ready にせず blocked を返してください。

- constraints / description / 完了意図の間に明確な矛盾がある
- 成功条件が曖昧で、implementer が安全に着手できない
- context_files 不足により主要判断ができない
- 実在確認できないファイル・skill・前提に依存している
- 未確定事項を無理に固定しないと routing できない
- 同時成立が難しい条件があり、task として実行可能性が低い

## blocked 時の出力ルール
- status は "blocked" にしてください
- skill_selection_reason には以下を具体的に書いてください
  - どの constraints または description の記述が矛盾・不足・未確定の原因か
  - 具体的にどのような矛盾が発生しているか（例: 「制約A と 制約B が共存できない」）
  - 実在確認できないファイル・skill・前提がある場合はその名称
- initial_execution_notes には以下を具体的に書いてください
  - 何を修正・確認すれば ready にできるか（例: 「制約X を削除する」「context_files に Y を追加する」）
  - constraints をどのように整理・分割すれば矛盾が解消されるか
  - ready にするために決定が必要な未確定事項の具体的なリスト
- blocked でも used_skills には "route-task" を入れてください
- blocked の理由が複数ある場合はすべて列挙してください
- 空の配列や曖昧な一言で済ませず、次に人または後続工程が判断できる材料を残してください
- blocked の場合でも docs_update_plan は必ず埋めてください
- blocked 原因の整理自体を docs に残す価値が低いなら docs_update_plan.needs_update は false でよいです

## carry_over 処理ルール

- `context_files` に前 task の `director_report_v{N}.json` が含まれる場合、そのファイルの `remaining_risks` フィールドを読み込み、`initial_execution_notes` の **先頭** に `[carry_over from {task_id}]` プレフィックス付きで転記すること
- `remaining_risks` が空配列の場合は転記不要
- 転記する際は元の `remaining_risks` の文言を改変しないこと（意図のずれを防ぐため）
- 転記形式: `"[carry_over from TASK-XXXX] {remaining_risks の内容}"`

## 重要な実行指示
- task_router report JSON を作成するだけでなく、必ず指定された保存先にファイルとして保存してください
- 保存先は必ずこの task の対象 repo 配下の .claude_orchestrator を使用してください
- 親repoや別repoの .claude_orchestrator に保存してはいけません
- ファイル保存が成功してから応答を返してください

## output rules
- JSON のみを返してください
- コードフェンスは禁止です
- 前置き文は禁止です
- role は "task_router" にしてください
- task_id と cycle は入力と一致させてください
