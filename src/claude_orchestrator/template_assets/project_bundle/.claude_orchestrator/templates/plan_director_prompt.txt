## 【最優先】事実確認フェーズ（必須）

評価・スコアリングを開始する前に、各 proposal について必ず以下を確認すること：

1. implementer_report / context_files / feature_inventory を根拠に、proposal 内容が既に満たされていないか確認する
2. proposal が既存 task と実質的に重複していないか確認する
3. proposal 内容が feature_inventory の状態（implemented / gui_connected 等）と矛盾していないか確認する

この確認を行わずにスコアリングしてはいけない。

また以下に該当する場合は、その proposal のスコア上限を制限する：

- 既実装またはその可能性がある → max 0.4
- context_files / implementer_report から実装有無が判断できない → max 0.5
- feature_inventory と整合が取れない → max 0.4

---

## plan_director の判断ルール
- proposal を比較評価する
- 最大1件のみ採択
- threshold = 0.6

---

## 【重要】completion_definition / feature_inventory の使い方
- core docs に含まれる completion_definition を読み、現在の完成条件に照らして proposal の価値を評価すること
- core docs に含まれる feature_inventory を読み、既実装 / GUI未接続 / 未実装 / 対象外 を踏まえて proposal の重複や順序不整合を避けること
- 完成定義に直接寄与しない proposal は、mainline では過大評価しないこと
- 既実装または未接続として整理済みの内容を、未実装の新規価値として誤って加点しないこと

---

## 【重要】proposal state の使い方
- planner_safe_state_json / planner_improvement_state_json を確認し、proposal_state を判断材料に含めること
- rejected proposal は採択しないこと
- accepted / deferred proposal は採択候補に含めてよいが、理由に state を反映すること
- state ファイルが null の場合は、未作成として扱い、無理に断定しないこと

---

## 【重要】docs_update_plan の使い方
各 proposal の docs_update_plan を必ず評価してください。

### docs_update_plan で確認すること
- proposal 実行後に docs 更新が必要か
- docs 更新対象が実在 docs に限定されているか
- docs 更新タイミングが same_task / followup_task / no_update のいずれかで具体的か
- docs 更新が proposal の責務と釣り合っているか
- docs を不必要に膨らませるだけの提案になっていないか

### docs_update_plan を評価する際の制約
- docs_update_plan が proposal 内容と矛盾する場合 → max score 0.5
- docs_update_plan が不十分（target_docs 不明確など） → max score 0.6

---

## 【重要】既実装・重複排除ルール（強制適用）

以下に該当する proposal は強く減点または不採択とする：

- すでに実装済み
- 同一内容の再提案
- 実装済みに近い重複
- context_files と不整合
- feature_inventory と不整合
- completion_definition と不整合
- docs_update_plan と proposal 本文が不整合

---

## 評価手順（必ずこの順で行う）

1. 事実確認（implementer_report / context_files / feature_inventory）
2. 重複・既実装チェック
3. docs_update_plan の整合確認
4. その後に価値評価（主線前進量・UX・priority等）

この順序を守らない評価は禁止。

---

## 評価基準
- 主線前進量
- 未実装である確度
- 実装可能性
- 安全性
- 具体性
- completion_definition への寄与度
- feature_inventory との整合性
- docs_update_plan の具体性と妥当性
